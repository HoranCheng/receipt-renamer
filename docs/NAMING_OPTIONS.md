# Naming Options — Receipt Renamer

## Background

命名格式已经来回改过一次，不能再擅自改。

历史：

- v1：日期 + 分类
- v2：日期 + 商户 + 金额
- 当前老板倾向：回到“日期 + 分类”方向

所以这里不给“拍脑袋结论”，只给清楚的选项和推荐。

---

## Candidate options

### Option A — 日期 + 分类

Example:

`2026-03-12_餐饮`

**优点**
- 最简洁
- 最适合快速扫一眼
- 对 OCR 精度要求最低
- 商户识别错了也不影响文件名

**缺点**
- 同一天同分类容易重名
- 后续找具体商户不方便

**适合**
- 老板最重视“看着清爽”
- 主要靠 Sheets 查详细信息

---

### Option B — 日期 + 分类 + 金额

Example:

`2026-03-12_餐饮_$45.60`

**优点**
- 仍然简洁
- 重名概率明显下降
- 金额是核心信息，检索价值高

**缺点**
- 比 A 稍长
- OCR 金额识别错误会直接体现在文件名里

**适合**
- 想在“简洁”和“唯一性”之间平衡

---

### Option C — 日期 + 分类 + 商户

Example:

`2026-03-12_餐饮_Woolworths`

**优点**
- 人类可读性更强
- 看文件名就知道是哪家

**缺点**
- 商户名 OCR 最容易错
- 文件名长度增长明显
- 商户别名规范化会变成长期维护问题

**适合**
- 老板强烈依赖“商户名快速浏览”

---

### Option D — 日期 + 分类 + 商户 + 金额

Example:

`2026-03-12_餐饮_Woolworths_$45.60`

**优点**
- 信息最全
- 重名概率最低

**缺点**
- 最长
- OCR 任一字段错都会显得难看
- 手机上看列表容易拥挤

**适合**
- 极度追求信息完整，不介意冗长

---

## My recommendation

### 推荐：Option B

`日期 + 分类 + 金额`

原因很直接：

1. 比 A 更不容易重名
2. 比 C / D 更稳，因为金额通常比商户名更可靠
3. 仍然保持文件名短
4. 真正的详细信息本来就该放在 Sheets，而不是全塞进文件名

---

## Tie-break rule

如果老板仍然偏爱极简，我建议这样决策：

- **优先美观** → A
- **优先实用** → B

我个人不建议默认走 C / D，除非老板明确说“必须看到商户名”。

---

## Implementation note

一旦老板选定：

1. 统一更新 rename 逻辑
2. 补一条 migration/兼容策略说明
3. 不再反复切格式

否则历史文件名会越来越乱。
