## 1. 执行摘要

### 当前产品阶段
这不是“概念原型”了，也还不是“可放心放给普通用户长期依赖的成熟工具”。更准确地说，它现在处于：**功能主链已跑通的内测 Beta**。

证据很明确：
- 主链路已经完整：上传/拍照 → Drive 入箱 → AI 识别 → 命名 → 分流到已存档/待确认 → 高置信度写入 Sheets（`src/views/ScanView.jsx`, `src/services/processor.js`, `src/views/ReviewView.jsx`）。
- 已经有多设备配置同步、失败重试、离线待传、Service Worker 协作、用户作用域隔离，而不是单页 demo（`src/App.jsx`, `src/services/google.js`, `src/services/storage.js`, `src/services/swBridge.js`, `src/services/pendingQueue.js`）。
- 但同时，文档、测试和部分旧视图没有跟上当前实现，说明产品在高速迭代中，工程一致性还没收口。

### 最强项
**最强项不是 UI，而是“围绕真实风险做过补洞”的主链路意识。**

具体体现：
- 不是盲目全自动，而是明确用了 `confidence` 分流，低置信度进入 Review，高置信度才自动归档（`src/services/processor.js`）。
- 对“AI 误判非小票”做了容错：如果 `is_receipt=false` 但仍提取出 merchant/amount/date，会强制降级到 review，而不是直接丢弃（`src/services/processor.js`）。这点非常关键，说明团队已经在修“信任断裂点”，不是只追求自动化表面完成率。
- 上传队列做了持久化与自动恢复，且对 Wi‑Fi-only、存储逼近上限、陈旧待传做了用户提醒（`src/views/ScanView.jsx`）。这比 PDF 里假设的“localStorage 原型工具”成熟得多。
- 配置支持云端同步与冲突解决，并且在冲突时把“云端 vs 本地”明确给用户选，而不是静默覆盖（`src/App.jsx`）。

### 最大风险
**最大风险已经不是“功能不够”，而是“系统存在两套现实”：运行时主链路在进化，但文档、测试、部分旧入口仍停留在旧架构。**

这会带来三个直接后果：
1. 产品负责人会被过时文档误导。`docs/ARCHITECTURE.md` 仍写着旧命名格式、旧 AI/存储表述，和现在代码不一致。
2. 团队会被过时测试误导。当前 `npm test` 实测失败，失败项集中在 `ai.test.js` 与常量测试，反映出代码已切到 proxy 模式和新文件夹命名，但测试没同步。
3. 用户路径存在“双轨制”。`ScanView + processor.js` 是新版主链路，但 `InboxView.jsx` 还保留一套旧的单独处理流程，能力和容错都更弱，后续很容易出现维护分叉。

如果只让我给一个最该盯的风险：**不是 AI 准确率，而是工程事实不一致导致的错误判断与回归。**

## 3. 基于真实代码的发现

## 3.1 UI / UX

### 做得好的
1. **扫描页已经很接近真实使用场景，而不是展示页。**
   - 拍照主按钮突出；
   - 相册导入支持多张；
   - Wi‑Fi-only 阻塞、缓存逼近上限、恢复未完成上传，都有明确视觉反馈（`src/views/ScanView.jsx`）。
   - 这比 PDF 里那种“可能还是开发者工具感很重”的判断更乐观。

2. **Review 页的信任表达是够用的。**
   - 明确展示 review reason、confidence、原图、Drive 链接；
   - 非小票状态有单独 badge；
   - 用户可手动修字段再通过（`src/views/ReviewView.jsx`）。

3. **配置页已经相当克制。**
   - 外层先讲用户能懂的概念：Google 账号、上传设置、Drive 文件夹、记录表、图片压缩、数据备份、危险区域；
   - 高级设置折叠；
   - 核删除设计认真，不轻浮。

### 仍然存在的问题
1. **Review 的修正效率仍偏“单张深处理”，而不是“批量快审”。**
   当前流程是：进入详情 → 看图 → 改字段 → 通过。单张可以，批量就会变慢。PDF 对“校正效率”担心没有错，只是它低估了现有 UI 质量、高估了它的严重程度。

2. **字段级不确定性没有可视化。**
   用户看到的是整单的 `confidence` 与 `reviewReason`，但看不到“日期最不准”还是“商户最不准”。这会让审核动作仍偏全量复查。

3. **Log 页和 Review 页的设计语言在继续分化。**
   - Log 页偏轻量、摘要化；
   - Review 页偏重型、工作台化；
   - 这本身没问题，但如果以后再加快审模式，最好避免第三种完全不同的交互风格。

4. **旧入口 `InboxView.jsx` 的交互标准落后于新版主链。**
   它保留了老式“列表逐个处理/一键全部处理”的模式，但没有新版 `processor.js` 那套更完整的容错和状态回传。这不是用户当下最痛的点，但会在后续演化中变成 UX 一致性问题。

## 3.2 核心业务链路

### 做得好的
1. **主链路比很多同类 MVP 更扎实。**
   `processor.js` 里把以下几件事真正串起来了：
   - 取文件 base64
   - 调 AI proxy
   - 处理 false negative
   - 用统一命名规则生成文件名
   - 高置信度直接进 validated，低置信度进 review
   - validated 才写 Sheets
   - Sheets 失败进入 outbox 重试

2. **“AI 说不是小票，但其实提到了交易数据”这个兜底非常关键。**
   这是最值得肯定的设计之一。很多产品在这一步直接把文件打成垃圾；这里选择低置信度送审，更符合真实世界票据脏数据分布。

3. **命名规则已经收敛到更稳的格式。**
   当前命名是 `YYYY-MM-DD_category_seq.ext`（`src/utils/naming.js`），并会从现有文件名播种计数器避免冲突。这比把商户名、金额直接塞进文件名更稳，也更抗 OCR 波动。

### 真问题
1. **存在两套处理链：`processor.js` 与 `InboxView.jsx`。**
   这是本次代码审最明确的架构问题之一。
   - `ScanView` 走的是新版队列 + `enqueueFile/processInboxBackground`；
   - `InboxView` 自己又实现了一套 `processFile/batchProcess`。

   两套逻辑的风险：
   - 行为不一致；
   - 某些修复只落一边；
   - 测试覆盖会越来越碎。

   这不是立刻炸用户的 P0，但它是非常典型的 P1：**会持续制造回归。**

2. **审核通过写 Sheets 时，记录对象不完整。**
   `ReviewView.jsx` 里 `onReceiptProcessed` 传回的对象没有带 `id/fileId/driveId`，而 `App.jsx` 的 `addReceipt` 逻辑也主要为“sheetSyncFailed 时本地兜底”服务。这说明 review→log 的本地桥接不是主真源，当前更多依赖 Sheets 读取刷新。这能用，但边界不够漂亮。

3. **文件命名与旧文档完全失配。**
   现在代码已经是 `date_category_seq`，但 `docs/ARCHITECTURE.md` 还写着 `YYYY-MM-DD_商户名_金额`。这会直接误导任何后续产品讨论、测试设计和用户文档。

## 3.3 数据与同步 / 存储

### 做得好的
1. **“云端真源 + 本地缓存/待同步层”的分层已经成型。**
   - 文件：Drive
   - 记录：Sheets
   - 配置：Drive config file
   - 本地：队列、恢复、作用域缓存

   这比 PDF 中“主要是 localStorage 风险”的判断成熟很多。

2. **用户隔离是认真做了的。**
   `storage.js` 用 `rr-current-user` 做 scoped key；`App.jsx` 在恢复配置时也尽量恢复 user scope。对于共享设备/切账号，至少做到了基本隔离。

3. **配置冲突处理是可理解的。**
   `App.jsx` 会识别本地新设备 vs 已自定义设备，并在冲突时显式让用户选“用云端的 / 用本地的”，而不是偷偷覆盖。

4. **Sheets 失败 outbox 是对的。**
   票据系统最怕“看起来成功，实际上没记账”。这里至少做了失败记录和重试。

### 真问题
1. **存储层命名与实现仍然有“历史包袱”。**
   文档里写 IndexedDB 抽象，但 `storage.js` 实际是 localStorage 封装；而 `pendingQueue/imageCache` 又是另一套 IndexedDB。也就是说，系统不是没有分层，而是**分层事实没有被准确表达**。

2. **`clearAllData()` 的范围很猛，依赖调用方自己做字段保留。**
   现在 `App.jsx` 的 reset 逻辑是先保存 auth 相关字段，再调用 `clearAllData()`，再写回保留字段。能工作，但耦合较高，后续若新增关键配置字段，很容易漏保留。

3. **`saveCloudConfig()` 吞错。**
   `src/services/google.js` 中保存云端配置失败只 `console.warn`，对用户层面基本无感知。对“多设备配置同步”这种能力，静默失败是危险的，因为用户会以为跨设备一致。

## 3.4 错误处理与信任设计

### 做得好的
1. **对真正会伤信任的场景，已经有不少具体设计。**
   - 上传超时按文件体积分档；
   - Wi‑Fi-only 阻塞有明确 banner；
   - 本地缓存逼近上限有告警；
   - 非小票有独立提醒；
   - 登录过期在 Review 加载时会告诉用户去设置页重连；
   - 核删除有清晰边界说明。

2. **不是所有错误都“算失败”，有分层。**
   例如：
   - AI 判断非小票但有数据 → review，不算粗暴失败；
   - Sheets 失败 → receipt 仍可 validated，但标记 `sheetSyncFailed` 并进 outbox。

### 真问题
1. **“配置同步失败”与“部分云端动作失败”的用户可见性不够。**
   这类错误目前很多停留在 console warn。对内测团队还行，对产品负责人判断稳定性不够透明。

2. **Log 页删除与数据真源之间的语义值得再核对。**
   当前 `LogView` 数据主源来自 Sheets，但其删除回调接的是本地 `receipts` 删除函数（`App.jsx` 中 `deleteReceipt`）。这意味着：
   - 如果当前展示的是 sheetRecords，左滑删除未必真删到云端真源；
   - 详情页编辑也是改本地 receipt，而不是直接回写 Sheets。

   这不是我建议立刻返工的大坑，但它说明：**Log 现在更像“查看器 + 本地残留兜底”，还不是完整的云端记录管理器。** 如果产品负责人把它理解成“正式账本编辑页”，那会高估当前能力。

3. **有一部分信任设计仍停留在文案级，不是结构级。**
   比如用户仍然看整单 confidence，而不是字段可信度；看到“需要核查”，但不一定知道先看哪里。

## 3.5 测试与工程质量

### 做得好的
1. **不是没测，且测到了真正容易出事故的边界。**
   `processor-recognition.test.js` 的覆盖质量比我预期好，包含：
   - 置信度边界值；
   - false negative override；
   - $0 金额；
   - PDF；
   - 外语票据；
   - Sheets 重试与 outbox。

2. **命名规则测试也够扎实。**
   `src/utils/__tests__/naming.test.js` 覆盖了 seed、递增、老格式忽略、分类 slug 等关键点。

### 真问题
1. **当前测试套件是红的，而且红得不是小毛病。**
   我实跑 `npm test`，结果失败集中在：
   - `src/services/__tests__/ai.test.js` 5 个全挂；
   - `src/constants/__tests__/index.test.js` 1 个挂。

   这说明当前仓库至少存在两类问题：
   - 测试已明显落后于现在的 AI proxy 实现（`src/services/ai.js` 现在只负责 proxy 请求和输入校验，不再做旧版解析逻辑）；
   - 常量测试仍期待旧文件夹名 `00_inbox / 10_validated`，但实际默认配置早已换成中文命名。

   换句话说：**当前“有很多测试”不等于“当前代码处于受保护状态”。**

2. **文档也明显过期。**
   - `docs/ARCHITECTURE.md` 仍写 Gemini、旧命名、旧数据表述；
   - 用户要求重点阅读的 `docs/DEPLOY.md` 在仓库里不存在。

   这不是格式问题，而是会影响团队决策的问题。

3. **版本感知混乱。**
   `package.json` 还是 `0.1.0`，文档写 `v0.3.0`，而实际功能成熟度已经明显高于这两个标识之一。对外可以慢，但对内最好统一。

## 4. 只列真正高优先级的问题（P0 / P1 / P2）

## P0

### P0-1 测试与当前实现脱节，主干测试红灯
**为什么是 P0：**
这会直接破坏后续迭代的判断基础。不是“覆盖率低”，而是“测试在告诉你旧世界的真相”。

**证据：**
- `npm test` 失败；
- `ai.test.js` 仍按旧 AI 调用/响应格式断言；
- `constants` 测试仍按旧 Drive 文件夹命名断言。

**建议：**
- 先把测试分成“当前有效”与“历史作废”两类；
- 优先修复 `ai.test.js` 和常量测试，让 CI 至少回到绿灯；
- 之后再补 proxy 层的真实契约测试。

### P0-2 文档严重过期，且关键交付文档缺失
**为什么是 P0：**
对这个项目来说，产品决策高度依赖文档；过时文档会比没文档更危险。

**证据：**
- `docs/ARCHITECTURE.md` 与当前命名、AI 路由、数据分层明显不符；
- `docs/DEPLOY.md` 缺失。

**建议：**

### P1-3 配置同步失败的用户感知不足
**为什么是 P1：**
跨设备是信任能力，不能长期停留在 console.warn。

**建议：**

## 6. 给 Claude 的讨论要点

### 我最坚持的 3 个结论
1. **这个产品已经不是“原型”，而是内测 Beta。** 再按“只有概念正确”去评，会低估它已经做出来的脏活能力。
2. **现在最大风险不是 AI 准确率，而是工程事实不一致：代码在前，测试和文档在后。** 这会直接扭曲后续所有判断。
3. **最该收口的是统一主链，而不是大拆组件。** `processor.js` 与 `InboxView.jsx` 双轨并存是后续回归源头。

### 我认为最容易被高估的点
1. **“测试不少，所以工程很稳。”** 不对。当前测试是红的，而且红在关键契约层。
2. **“有云端真源，所以数据层已经很成熟。”** 也不对。配置同步与 Log 页语义还没完全产品化。
3. **“UI 已经做得不错，所以核心体验问题不大。”** UI 的确比 PDF 说的成熟，但 Review 的批量效率问题仍然存在。

### 我认为最容易误判的点
1. **误判为“仍以 localStorage 为主的数据架构”。** 当前事实已经不是这样。
2. **误判为“设置页还是纯开发者界面”。** 这条已部分修复，不该继续沿用旧批评强度。
3. **误判为“最优先应该做大重构”。** 当前最优先其实是修测试、修文档、收敛双链路。

## 结论（给产品负责人）
这个项目最可贵的地方，不是它已经把 UI 做到多精致，而是它已经从“演示自动识别”往“处理真实票据风险”迈出了一大步。当前代码显示出明确的产品判断：知道 AI 会错、知道网络会断、知道用户会切设备、知道本地队列会堆积、知道删除是高风险动作。

但下一阶段不能再继续“功能先跑、文档和测试以后补”。因为它现在已经不是可以靠脑内模型维护的小项目了。**从这一刻开始，最大的产品风险就是工程现实分叉。**

我的建议很简单：
- 先把当前世界说准（文档）；
- 再把当前世界测准（测试）；
- 再把当前世界做成一套（统一处理链）；
- 最后再去追求更快的审核体验和更漂亮的界面。

如果这三步不先做，后面每多加一个功能，都会让产品负责人更难判断“现在到底稳不稳”。
