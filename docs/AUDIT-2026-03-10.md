# Receipt Renamer 审计报告

**项目：** receipt-renamer + receipt-proxy  
**日期：** 2026-03-10  
**审计范围：** 代码、线上站点、Cloudflare Worker 暴露面、测试状态、市场竞品、产品方向  
**结论摘要：** 产品方向正确，体验设计有亮点，但当前版本仍属于 **适合内测 / 小范围使用** 的状态，距离“可放心公开放量”还差一轮安全加固与可靠性补强。

---

## 1. Executive Summary

### 总判断

这个产品现在的状态可以概括为：

- **产品方向：对**
- **用户体验：很多关键判断是对的**
- **技术架构：MVP 合理**
- **安全壳层：还不够硬**

### 最大风险

当前最值得优先处理的，不是识别效果，而是：

1. **Worker 可被滥用，直接造成 Gemini API 成本损失**
2. **Quota 机制可以被绕过**
3. **前端缓存与 token 持久化存在隐私 / 安全放大风险**
4. **Drive / Sheets 之间缺少一致性保障，可能导致账目静默缺失**

### 适合当前定位

- 适合继续快速迭代
- 适合你自己和少量熟人内测
- **不建议在修掉高危项之前放量公开传播**

## 4. 详细发现

## 4.1 Critical

### C-1 Worker 信任前端裸传 `uid`

**相关文件：**
- `receipt-proxy/src/index.js`
- `receipt-renamer/src/services/ai.js`

**问题：**
当前 `/api/analyze` 和 `/api/quota` 接收浏览器传来的 `uid`，而不是服务端自己验证后生成的身份。

更严重的是，前端现在读取的是：

- `localStorage.getItem('receipt_google_uid') || 'anonymous'`

但项目里并没有一条真正稳定、可信的逻辑把 Google 用户真实 `sub` 写进去。

**后果：**
- 很多用户可能都落进同一个 `anonymous` 配额桶
- 攻击者可以随便伪造不同 uid 来无限刷额度
- 攻击者也可以用别人的 uid 去消耗别人配额
- 你的 Worker 没法区分“真实用户请求”还是“外部脚本滥用”

**风险性质：** 直接财产损失风险

**建议修复：**
- 不再信任前端裸传 uid
- 由 Worker 验证 Google 身份凭证后，自行取 `sub`
- 最低可行方案：前端传 Google ID token，Worker 验签后得出 uid
- 删除 `anonymous` fallback

---

### C-2 Worker 允许任意 Origin 跨域访问

**相关文件：**
- `receipt-proxy/src/index.js`
- `wrangler.toml`

**问题：**
CORS 当前是：

- `Access-Control-Allow-Origin: *`

意味着任何网页、任何脚本都可以把你的 Worker 当成公开接口来用。

**后果：**
- 别人可直接拿你的 Worker 做免费 OCR / AI 代理
- 配合可伪造 uid，额度形同虚设
- 你承担 API 成本

**建议修复：**
- 改成 Origin allowlist
- 至少只允许：`https://horancheng.github.io`
- 后续如有自定义域名，再加入新域名

---

### C-3 Worker 没有限制请求大小 / MIME / 文件类型

**相关文件：**
- `receipt-proxy/src/index.js`

**问题：**
当前只校验字段存在，不校验：
- 文件大小
- base64 长度
- MIME 白名单
- fileType 合法性
- PDF 页数 / 图像尺寸

**后果：**
- 可被超大 payload 打爆
- 可增加 Worker CPU / 内存压力
- 可推动 Gemini 产生不必要费用
- 可能导致服务变慢甚至不可用

**建议修复：**
- 允许 MIME 白名单：`image/jpeg` / `image/png` / `application/pdf`
- 图片和 PDF 各自设大小上限
- 超限直接拒绝，不进入 Gemini

---

## 4.2 High

### H-1 Quota 读写不是原子操作

**相关文件：**
- `receipt-proxy/src/index.js`

**问题：**
当前逻辑是：
1. 读取 used
2. 判断是否超限
3. 调 Gemini
4. 写回 `used + 1`

Cloudflare KV 不是为这种强一致计数设计的。

**后果：**
- 并发下可超发
- 配额统计不准
- 成本超过预期

**建议修复：**
- 用 Durable Objects 做每用户每日原子计数
- 或引入可串行化的配额控制机制

---

### H-2 Debug 接口公开暴露

**相关文件：**
- `receipt-proxy/src/index.js`

**问题：**
`/api/debug/gemini` 在线上仍可访问。

**暴露信息：**
- keyHint
- upstream 可用性
- models 列表

**后果：**
- 给攻击者做探测提供情报
- 暴露部署形态

**建议修复：**
- 线上删除该接口
- 或加管理员密钥保护

---

### H-3 生产环境保留危险的浏览器直连 AI fallback

**相关文件：**
- `receipt-renamer/src/services/ai.js`

**问题：**
如果 `VITE_AI_PROXY_URL` 缺失，会 fallback 到浏览器直连 Anthropic。

而 Vite 的 `VITE_*` 变量是会被打包到前端的。

**后果：**
- 一次错误部署，就可能把付费 API key 暴露给所有访问者

**建议修复：**
- 生产环境 fail closed
- 没配 proxy 时直接报错，不允许 fallback
- 如果保留直连，只允许本地开发环境使用

---

### H-4 Google access token 持久化到 Web Storage

**相关文件：**
- `receipt-renamer/src/services/google.js`

**问题：**
Access token 现在存在：
- `sessionStorage`
- 可选 `localStorage`

**后果：**
若未来出现：
- XSS
- 恶意浏览器扩展
- 共享设备被他人打开

则可能直接获得用户 Drive / Sheets 权限。

**建议修复：**
- 优先只存内存
- 若必须持久化，缩短生命周期
- 增加 CSP
- “记住我” 明确标注不适合共享设备

---

### H-5 自动命名部分路径仍然用 category

**相关文件：**
- `receipt-renamer/src/services/processor.js`

**问题：**
目前 processor 中自动归档路径仍然存在：
- `YYYY.MM.DD Category Amount.ext`

而不是你定义的：
- `YYYY.MM.DD MerchantName Amount.ext`

**后果：**
- 核心业务逻辑不一致
- 超市仍可能被泛化
- 文件检索价值下降

**建议修复：**
- 统一改用 merchant
- 且做 title-case + safeName

---

## 4.3 Medium

### M-1 Service Worker 缓存策略过宽

**相关文件：**
- `receipt-renamer/public/sw.js`

**问题：**
当前会缓存所有 GET 请求。

**后果：**
- 可能缓存私有 Google 响应
- 用户退出后仍残留数据
- 缓存污染
- PWA 行为不稳定

**建议修复：**
- 只缓存 same-origin 静态资源
- API / Drive / Sheets / OAuth 一律不要缓存

---

### M-2 sign out / reset 未清空本地敏感缓存

**相关文件：**
- `src/App.jsx`
- `src/services/imageCache.js`
- `src/services/pendingQueue.js`

**问题：**
现在 reset / sign out 主要清：
- config
- receipts
- session

但没完全清：
- IndexedDB 原图缓存
- 待上传队列
- non-receipt alerts

**后果：**
共享设备上会残留敏感小票图。

**建议修复：**
统一实现：
- `clearAllLocalSensitiveData()`

---

### M-3 Sheets 写入失败会静默丢账

**相关文件：**
- `src/services/processor.js`
- `src/views/ReviewView.jsx`

**问题：**
Drive move 成功后，Sheet append 失败经常只是 warning/catch。

**后果：**
- 文件归档成功
- 表格没写进去
- 用户不一定知道

**建议修复：**
- 增加 outbox / retry 机制
- 记录 `sheet_pending`
- 在 UI 中提示未同步条目

---

### M-4 浏览器端 base64 转换过于吃内存

**相关文件：**
- `src/services/google.js`

**问题：**
大图片 / PDF 会让移动端内存压力非常大。

**建议修复：**
- 限制上传尺寸
- 优化转换方式
- 长期考虑改成 worker 端取文件

---

### M-5 remember me 状态流不完整

**问题：**
rememberMe 配置来源分散，登录和恢复逻辑不完全统一。

**后果：**
- 用户感知不稳定
- “记住我” 行为偶发失效

**建议修复：**
- 把 rememberMe 作为 config 的单一真相源

---

## 5. 线上站点 / 部署观察

### 发现 1：Service Worker 注册路径风险
线上站点跑在：
- `/receipt-renamer/`

因此 service worker、manifest、icon 等资源路径必须始终与 base 对齐。

### 发现 2：GitHub Pages 适合 MVP，但安全控制能力有限
优点：
- 免费
- 快速部署
- 很适合当前阶段

限制：
- Header 级安全策略控制有限
- 更依赖前端代码自律

结论：
- 作为 MVP 合理
- 若未来公开放量，可考虑自定义域名 + 更细致的边缘控制

## 6.2 你的优势

### 1. 切口对
你抓的不是“OCR 本身”，而是：

> 小票识别之后，如何真正变成可管理的文件资产。

### 2. 用户数据 ownership 强
- 文件在用户自己的 Google Drive
- 用户更安心
- 迁移成本低

### 3. 异步流程设计正确
- 拍照后不阻塞用户
- 后台识别非常符合移动端真实场景

### 4. Review queue 思路成熟
- 高置信度自动过
- 低置信度再人工处理
- 让 AI 真正节省用户时间

### 5. 中文用户体验好
这在同类产品里是一个明显优势。

## 8. 建议修复顺序

## 第一优先级：先保命
1. 删除 `/api/debug/gemini`
2. Worker 不再信任前端裸传 `uid`
3. 增加 body size 限制 + MIME allowlist
4. 收紧 CORS origin
5. 增加真正的 quota / rate limit 防护
6. 修 processor 命名逻辑（category → merchant）

## 第二优先级：补可靠性
7. sign out / reset 清理所有本地敏感缓存
8. 修 service worker 注册与缓存策略
9. 给 Sheets 增加补偿 / retry / 未同步提示
10. 修 remember me 状态流
11. 修测试

## 第三优先级：拉开产品差距
12. merchant normalization / alias memory
13. 更强的 review queue
14. analytics / 月度洞察
15. tax / warranty / return 等高价值场景

## 9. 最终结论

**值得。**

**不建议。** 至少在高危项修掉之前，不建议大范围传播。

### 最准确的评价

也就是说：
- 这是一个 **很有潜力的产品**
- 不是方向错了
- 而是当前需要做一轮 **“从能用 → 能放心用”** 的升级
